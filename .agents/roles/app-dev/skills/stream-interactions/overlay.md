# Overlay

## What It Is

The overlay (`packages/overlay/`) is a Phaser 3 browser application that runs as a single OBS browser source. It connects to the broker via WebSocket, subscribes to relevant topics, and renders stream events as visual overlays on top of the live scene â€” alerts, animations, character moments, and any other time-sensitive display.

---

## OBS Integration Model

The overlay is a single browser source in OBS. It occupies the full 1920Ă—1080 canvas with a transparent background, sitting in the OBS layer stack above the stream content.

**Two modes:**

| Mode | OBS browser source URL | When to use |
|---|---|---|
| Development | `http://localhost:5173` | Testing, iterating â€” Vite dev server with HMR |
| Production | Local file: `dist/index.html` | Live stream â€” built output, no server dependency |

To switch modes, change the OBS browser source URL. The built output uses `base: './'` in `vite.config.ts` so all assets resolve correctly as local files.

**Build for production:**
```
cd Apps/stream-overlay
pnpm build
```
OBS then loads `packages/overlay/dist/index.html` directly.

## The Overlay is a Dumb Renderer

This is the most important design constraint:

- âś… The overlay renders what it's told
- âś… It subscribes to broker topics and maps topics to Phaser scenes/animations
- âťŚ It does NOT contain business logic â€” it doesn't decide *when* to show something
- âťŚ It does NOT maintain authoritative stream state (squad status, chaos meter values)
- âťŚ It does NOT talk to Streamer.bot, Twitch, or Mix It Up directly

If you find yourself writing "if squad size > 3, show the big alert" in the overlay, stop. That decision belongs in Streamer.bot. The overlay should receive a message that says "show big alert" and render it.

## Broker Connection

On startup, the overlay:
1. Connects to the broker via WebSocket (`BROKER_URL` from `@stream-overlay/shared`)
2. Sends `ClientHello` identifying itself as `CLIENT_NAMES.OVERLAY` (`"overlay"`)
3. Declares its topic subscriptions (see below)
4. Listens for `BrokerMessage` deliveries and delegates to the appropriate Phaser scene

Disconnection and reconnection logic is handled in `broker-client.ts`.

---

## Topic Subscriptions

The overlay subscribes to the following topic namespaces in its `ClientHello`:

| Wildcard | Covers | Why |
|---|---|---|
| `overlay.*` | `overlay.spawn`, `overlay.move`, `overlay.animate`, `overlay.remove`, `overlay.clear` | Core asset rendering commands |
| `overlay.audio.*` | `overlay.audio.play`, `overlay.audio.stop` | Sound playback control |
| `lotat.*` | All `lotat.*` topics | LotAT session UI â€” join counter, node text, vote tally, chaos meter, dice/commander overlays |
| `squad.*` | All squad game topics, all games | Squad game visual reactions |
| `stream.*` | Subs, raids, follows, bits | Alert animations |
| `system.*` | Connect/disconnect, ping/pong | Connection health |

**ClientHello subscriptions array (v1):**
```typescript
subscriptions: [
  TOPICS.WILDCARD_OVERLAY,        // "overlay.*"
  TOPICS.WILDCARD_OVERLAY_AUDIO,  // "overlay.audio.*"
  TOPICS.WILDCARD_LOTAT,          // "lotat.*"
  TOPICS.WILDCARD_SQUAD,          // "squad.*"
  TOPICS.WILDCARD_STREAM,         // "stream.*"
  TOPICS.WILDCARD_SYSTEM,         // "system.*"
]
```

## Topics the Overlay Publishes

In v1, the overlay **publishes nothing** except:
- `system.ping` â€” sent on a keepalive interval to verify broker connection is alive

The overlay is a pure subscriber/renderer. It does not publish game state, votes, or commands.

## Scene Architecture

One permanent scene with four named layer groups â€” no scene swapping:

```
OverlayScene (always running)
  â”śâ”€â”€ alertLayer   â€” ephemeral stream alerts (subs, raids, follows, bits)
  â”śâ”€â”€ lotatLayer   â€” LotAT session UI (join counter, vote tally, chaos meter)
  â”śâ”€â”€ squadLayer   â€” squad mini-game visuals
  â””â”€â”€ hudLayer     â€” persistent HUD elements (highest depth)
```

**Why one scene:** The overlay is a dumb renderer â€” it never transitions between "modes." LotAT and a squad game can coexist on the canvas simultaneously. Layer groups provide z-ordering without scene lifecycle complexity.

**Adding new content types:** Create a new `Phaser.GameObjects.Group` in `OverlayScene.create()` and register broker subscriptions for its topic namespace. No scene swap needed.

Scene flow on startup:
```
BootScene.preload() â†’ (no preloading â€” all assets loaded on demand)
BootScene.create()  â†’ this.scene.start('OverlayScene')
OverlayScene.create() â†’ registers all broker topic subscriptions
```

## Broker Connection Behavior

- The `brokerClient` singleton starts connecting immediately when `main.ts` loads.
- Sends `ClientHello` with name `"overlay"` and all rendering-relevant wildcard subscriptions.
- Receives `ClientWelcome` confirming registration. Starts 15-second ping keepalive.
- If the broker is not running: silent retry with exponential backoff (1s â†’ 2s â†’ 4s â†’ ... â†’ 30s max). No crash, no visible error (unless `?debug=true`).
- If the connection drops mid-session: same backoff reconnect. The overlay renders an empty canvas until reconnected.
- On reconnect: a new `ClientHello` is sent â€” subscriptions are re-established automatically.

## Asset System

The overlay has a full asset management system built on three classes in
`src/systems/`:

| Class | Responsibility |
|---|---|
| `AssetManager` | Registry, spawn/move/animate/remove/clear |
| `AnimationSystem` | Phaser tween wrappers for all `AnimationPreset` strings |
| `AudioManager` | MP3 load-on-demand and playback via Phaser WebAudio |

**All visual assets â€” PNG, JPG, and GIF â€” are Phaser game objects.** GIFs use a
hidden `<img>` element approach: a native browser `<img>` animates the GIF,
each frame is drawn into a `Phaser.Textures.CanvasTexture` via `drawImage()` +
`refresh()` in `OverlayScene.update()`. The rendered object is a standard
`Phaser.GameObjects.Image`, so Phaser tweens work identically on GIFs and static
images. There is no split rendering model and no special cases for animated content.

See [`asset-system.md`](asset-system.md) for:
- Full asset lifecycle diagram
- All supported animation presets
- Asset folder conventions (`public/assets/{alerts,lotat,squad,audio}/`)
- Complete message reference with example payloads
- Manual test procedure (12 tests covering all message types)
