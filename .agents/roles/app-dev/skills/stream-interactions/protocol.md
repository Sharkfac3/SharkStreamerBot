# Message Protocol

## Authority

All types, constants, and payload shapes defined here are implemented in `@stream-overlay/shared`:

```
Apps/stream-overlay/packages/shared/src/
  protocol.ts   вЂ” TypeScript interfaces (BrokerMessage, ClientHello, all payloads)
  topics.ts     вЂ” TOPICS constants, BROKER_PORT, BROKER_URL, CLIENT_NAMES
  index.ts      вЂ” barrel export (re-exports both files)
```

**If this doc conflicts with the TypeScript source, the TypeScript source wins.**
Update this doc to match; do not patch types to match stale docs.

## Message Envelope

Every message through the broker вЂ” in both directions вЂ” must conform to `BrokerMessage<P>`:

```typescript
interface BrokerMessage<P = unknown> {
  id: string;        // UUID v4, generated by sender before publishing
  topic: string;     // dot-notation topic, e.g. "overlay.spawn"
  sender: string;    // stable client name: "streamerbot" | "overlay"
  timestamp: number; // Unix epoch milliseconds вЂ” Date.now()
  payload: P;        // topic-specific data вЂ” shape defined in protocol.ts
}
```

**Rules:**
- `id` is generated by the **sender**, not the broker. Use `crypto.randomUUID()`.
- `sender` must match the `name` sent in `ClientHello`. It is stable across reconnects.
- `topic` must exactly match a constant from `TOPICS` in `topics.ts`. Never hardcode strings.
- The broker does not validate `payload` shape вЂ” that is a TypeScript compile-time concern.
- Use `TypedBrokerMessage<"overlay.spawn">` for fully typed message construction.

## Client Handshake

Every client вЂ” Streamer.bot, the overlay, future web apps вЂ” uses the same two-step handshake:

```
Client                           Broker
  |                                |
  |--- WebSocket connect --------->|
  |--- ClientHello --------------->|   { type: "client.hello", name, subscriptions }
  |<-- ClientWelcome --------------|   { type: "client.welcome", clientId, connectedClients }
  |                                |
  |    [subscriptions active]      |
  |--- BrokerMessage (publish) --->|   broker routes to all subscribers of topic
  |<-- BrokerMessage (deliver) ----|   only if this client subscribed to the topic
  |                                |
  |--- WebSocket close ----------->|
  |                                |--- system.client.disconnected в†’ all subscribers
```

`ClientHello` and `ClientWelcome` are **not** `BrokerMessage` envelopes вЂ” they are raw handshake frames sent directly on the socket before the pub/sub loop begins.

## Topic Taxonomy

Topics use dot-notation namespaces. All strings are exported as `TOPICS.*` constants.

```
system.*          вЂ” broker infrastructure (connect, disconnect, ping/pong)
overlay.*         вЂ” generic canvas asset commands (spawn, move, animate, remove, clear)
overlay.audio.*   вЂ” sound playback commands
lotat.*           вЂ” LotAT session lifecycle events (mirrors the stage machine)
squad.*           вЂ” squad mini-game events (game-agnostic base)
squad.duck.*      вЂ” Duck mini-game events
squad.clone.*     вЂ” Clone mini-game events
squad.pedro.*     вЂ” Pedro mini-game events
squad.toothless.* вЂ” Toothless mini-game events
stream.*          вЂ” raw Twitch platform events (subs, raids, follows, bits)
```

## Wildcard Subscriptions

Clients subscribe at any granularity they need. Wildcards are **single-level** only.

| Wildcard Constant | Pattern | Matches |
|---|---|---|
| `WILDCARD_SYSTEM` | `system.*` | All system topics |
| `WILDCARD_OVERLAY` | `overlay.*` | All overlay asset topics (not audio) |
| `WILDCARD_OVERLAY_AUDIO` | `overlay.audio.*` | Audio play/stop |
| `WILDCARD_LOTAT` | `lotat.*` | All LotAT lifecycle topics |
| `WILDCARD_SQUAD` | `squad.*` | All squad topics, all games |
| `WILDCARD_SQUAD_DUCK` | `squad.duck.*` | Duck game only |
| `WILDCARD_SQUAD_CLONE` | `squad.clone.*` | Clone game only |
| `WILDCARD_SQUAD_PEDRO` | `squad.pedro.*` | Pedro game only |
| `WILDCARD_SQUAD_TOOTHLESS` | `squad.toothless.*` | Toothless game only |
| `WILDCARD_STREAM` | `stream.*` | All stream platform events |

**Single-level rule:** `squad.*` matches `squad.duck.start`. It does NOT match a hypothetical `squad.duck.rounds.1`. Subscribe to the narrower namespace explicitly if such nesting is added.

## Payload Shapes

Full TypeScript interfaces for every topic are in `protocol.ts`. Quick reference:

### Overlay Topics

**`overlay.spawn`** вЂ” Spawn an asset:
```typescript
{
  assetId: string;         // UUID v4 (ephemeral) or kebab-case slug (persistent)
  src: string;             // path relative to overlay/public/ or URL
  position: { x, y };     // canvas pixels (1920Г—1080 space)
  width?: number;
  height?: number;
  depth?: number;          // z-index; higher = on top
  enterAnimation?: AnimationPreset;
  enterDuration?: number;  // ms
  lifetime?: number;       // ms; omit for persistent assets
  exitAnimation?: AnimationPreset;
  exitDuration?: number;   // ms
}
```

**`overlay.move`** вЂ” Move existing asset:
```typescript
{ assetId: string; position: { x, y }; duration?: number; }
```

**`overlay.animate`** вЂ” Apply animation:
```typescript
{ assetId: string; animation: AnimationPreset; duration?: number; }
```

**`overlay.remove`** вЂ” Remove asset:
```typescript
{ assetId: string; exitAnimation?: AnimationPreset; exitDuration?: number; }
```

**`overlay.clear`** вЂ” Remove all (or filtered):
```typescript
{ assetIdPrefix?: string; exitAnimation?: AnimationPreset; exitDuration?: number; }
```

**`overlay.audio.play`** вЂ” Play sound:
```typescript
{ soundId: string; src: string; volume?: number; loop?: boolean; }
```

**`overlay.audio.stop`** вЂ” Stop sound:
```typescript
{ soundId?: string; }  // omit soundId to stop all
```

### Animation Presets

`"fade-in" | "fade-out" | "slide-in-left" | "slide-in-right" | "slide-in-top" | "slide-in-bottom" | "bounce" | "shake" | "pulse" | "spin" | "none"`

New presets must be supported in the overlay renderer before Streamer.bot can send them.

### Asset ID Strategy

| Asset type | ID format | Example |
|---|---|---|
| Ephemeral (alert, one-shot) | UUID v4 | `"3d2a1b0c-..."` |
| Persistent / named (HUD widget) | kebab-case slug | `"chaos-meter"`, `"lotat-hud"` |

Spawning an asset with an existing `assetId` replaces that asset. Named assets are a safe way to update persistent overlay elements without tracking UUIDs.

### LotAT Topics

LotAT topics map 1:1 to the runtime stages in `session-lifecycle.md`:

| Topic | Stage transition | Key payload fields |
|---|---|---|
| `lotat.session.start` | idle в†’ join_open | sessionId, storyId, title |
| `lotat.join.open` | join_open active | sessionId, windowSeconds |
| `lotat.join.update` | join_open, roster changed | username, participantCount |
| `lotat.join.close` | join_open в†’ node_intro (or ended) | participantCount |
| `lotat.node.enter` | node_intro | nodeId, readAloud, crewFocus, choices, diceHook?, commanderMoment? |
| `lotat.commander.open` | commander_open | commander, prompt, windowSeconds |
| `lotat.commander.close` | commander_open resolved | outcome, successText? |
| `lotat.dice.open` | dice_open | purpose, successThreshold, windowSeconds |
| `lotat.dice.roll` | dice_open, viewer rolled | username, rollValue, isSuccess |
| `lotat.dice.close` | dice_open resolved | outcome, outcomeText |
| `lotat.vote.open` | decision_open | choices, windowSeconds, participantCount |
| `lotat.vote.cast` | decision_open, vote received | username, voteTotals, votedCount |
| `lotat.vote.close` | decision_resolving or ended | winningCommand, resultFlavor, nextNodeId |
| `lotat.chaos.update` | any node entry with delta > 0 | chaosTotal, delta |
| `lotat.session.end` | ended в†’ idle | reason, endState? |

`lotat.node.enter` carries the full renderable node data (read_aloud, crew_focus, choices) so the overlay can display everything it needs from a single event. The mechanic sub-events (dice.open, commander.open) provide timing data for animated UI countdowns.

### Squad Topics

Squad payload shapes use generic envelopes with per-game typed state/result shapes:

```typescript
// squad.<game>.start
{ game: string; triggeredBy: string; }  // per-game start types extend this

// squad.<game>.update
{ game: string; state: Record<string, unknown>; }  // renderers cast state to typed shape

// squad.<game>.end
{ game: string; result: Record<string, unknown>; }  // renderers cast result to typed shape
```

Per-game state and result shapes are defined in `protocol.ts`:

| Game | Update state type | End result type |
|---|---|---|
| Duck | `SquadDuckUpdateState` | `SquadDuckEndResult` |
| Pedro | `SquadPedroUpdateState` | `SquadPedroEndResult` |
| Clone | `SquadCloneUpdateState` | `SquadCloneEndResult` |
| Toothless | *(no update event)* | `SquadToothlessEndResult` |

Renderers cast the generic `state` / `result` fields to the typed shape after reading the `game` discriminant. Do not add fields to squad payloads speculatively.

## Message Direction

```
Streamer.bot       publish      BROKER       deliver      Overlay
(Actions/ C#)  в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв–¶  (router)  в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв–¶  (Phaser)
                                  в”‚
                                  в””в”Ђв”Ђdeliverв”Ђв”Ђв–¶ Future Web App
                                  в””в”Ђв”Ђdeliverв”Ђв”Ђв–¶ Future AI Agent
```

| Publisher | Topics it publishes |
|---|---|
| Streamer.bot | `lotat.*`, `squad.*`, `stream.*`, `overlay.*` (render commands), `overlay.audio.*` |
| Broker | `system.client.connected`, `system.client.disconnected`, `system.pong` |
| Overlay (v1) | `system.ping` only вЂ” overlay is a pure subscriber in v1 |
| Future web app | TBD вЂ” likely player input on squad/lotat topics |

**The broker never generates application messages.** `system.*` events are the only messages the broker itself publishes.

## Error Handling

**v1 policy: silent drop + log.**

When the broker receives a message it cannot process (bad JSON, missing required fields, unknown topic):
- Log the failure with the sender's client ID, raw payload, and reason
- Drop the message вЂ” do not forward it, do not respond to the sender
- Never crash on a bad message

Rationale: all clients are trusted local TypeScript processes. Malformed messages should not occur in normal operation. If debugging demands error responses, add `system.error` in v2.

## Versioning Strategy

The protocol has no explicit version field in v1. Evolution rules:

| Change type | Strategy |
|---|---|
| Add an **optional** field to an existing payload | Safe вЂ” existing clients ignore unknown fields |
| Add a **new topic** | Safe вЂ” clients only receive what they subscribed to |
| **Remove** a field or topic | Breaking вЂ” requires coordinated update of all clients |
| Change a field's **type** | Breaking вЂ” requires coordinated update of all clients |
| Rename a topic | Breaking вЂ” treat as remove + add |

For breaking changes:
1. Add the new shape alongside the old (e.g. `overlay.spawn.v2`)
2. Migrate each client
3. Remove the old topic once all clients are updated

Do not add a `version` field to `BrokerMessage` until there is a concrete versioning need. Premature version routing adds complexity without benefit in a single-developer local service.
